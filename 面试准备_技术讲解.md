```

后端 FastAPI + 前端 Vue 3，通过 SSE 实现流式输出
```

## 二、整体架构（面试必讲）

### 2.1 请求链路图（要能画出来）

```
用户输入
   ↓
Vue 前端 → SSE 连接 → FastAPI Server (/chat/stream)
   ↓
Agent.run_stream()            ← 核心引擎
   ↓
ModelRouter.get_provider()    ← 路由到具体模型
   ↓
Provider.chat_stream()        ← 调用 API 流式返回
   ↓
   ├─ content token → 前端逐字显示
   ├─ thinking token → 思考链展示
   └─ tool_calls → 执行工具 → 结果送回模型继续
   ↓
完成 → 保存到 SQLite → 返回 done 事件
```

### 2.2 模块分层

| 层级 | 模块 | 职责 |
|------|------|------|
| 接入层 | `server.py` | FastAPI 路由、SSE 响应、文件上传 |
| 引擎层 | `agent.py` | Agent 主循环：消息构建 → 调用模型 → 工具执行 → 循环 |
| 路由层 | `router.py` | 多模型注册、路由、默认模型选择 |
| 提供层 | `providers/` | 各厂商 API 的适配封装 |
| 工具层 | `tools/` | 工具注册与执行（文件、计算、RAG、Web） |
| 知识层 | `rag/` | 文档分块、向量化、语义检索 |
| 持久层 | `db/` | SQLite 会话/消息/文件存储 |
| 记忆层 | `memory.py` | 对话上下文管理 |
| 扩展层 | `mcp/` | MCP 协议外部工具集成 |

## 三、核心模块深度讲解（面试重点）

### 3.1 多模型路由（Strategy 策略模式）

**文件：** [router.py](file:///f:/mini-agent/app/router.py)

#### 要讲的核心点：

**① 为什么用策略模式？**
- 每个模型厂商 API 格式不同（DeepSeek/Qwen/Kimi/GLM）
- 策略模式让每个 Provider 独立实现，互不干扰
- 新增一个模型只需新加一个 Provider 类，不用改现有代码

**② 关键实现：**
```python
# ModelRouter 维护一个 providers 字典
self._providers: dict[str, BaseModelProvider] = {}

# 通过 provider_name 统一 credential 读取
# 所有厂商共用 OpenAICompatibleProvider，只需配置不同的 model_id 和 base_url
```

**③ 动态注册机制：**
- Kimi 初始化时会从 API 拉取可用模型列表（`_register_kimi()`）
- 其他模型写死在 `_register_all()` 中
- 支持通过 `.env` 配置默认模型

**④ 兜底策略：**
```python
@property
def default_model_id(self) -> str:
    # 优先返回已配置 API Key 的模型
    # 如果配置的模型不可用，自动选第一个可用的
```

#### 面试话术：
> "这个项目需要支持多模型切换，我用了策略模式。每个厂商的 API 适配都封装在独立的 Provider 类里，通过 ModelRouter 统一管理。路由层面做了兜底——如果用户选的模型没配 API Key，会自动切换到第一个可用的模型，保证体验不中断。"

### 3.2 Agent 引擎（核心循环）

**文件：** [agent.py](file:///f:/mini-agent/app/agent.py)

#### 要讲的核心点：

**① Agent 主循环（`run_stream` 方法）**

```
循环最多 6-8 轮：
  1. 调用 provider.chat_stream() 获取流式事件
  2. 逐条 yield 给前端（token / thinking / tool_calls）
  3. 如果收到 tool_calls → 执行工具 → 结果追加到 messages → 继续循环
  4. 如果没有 tool_calls → 返回 done 事件 → 结束
```

**② 流式事件协议（设计考量）**
```
事件类型统一为 dict，保证前后端解耦：
  - {"type": "token", "content": "..."}       → 普通文本 token
  - {"type": "thinking", "content": "..."}    → 思考链 token
  - {"type": "status", "content": "..."}      → 工具调用状态提示
  - {"type": "reset", ...}                    → 工具返回后重置内容
  - {"type": "tool_calls", ...}               → 工具调用请求
  - {"type": "done", "content": "..."}        → 完成
  - {"type": "error", "content": "..."}       → 错误
```

**③ 消息构建（`_build_messages`）**
```
- 首条插入 system prompt（包含可用工具列表）
- 追加历史对话
- 如果用户上传了文件，将文件内容作为 system 消息注入
  （只取前 3000 字符，避免超出 token 限制）
```

**④ 工具调用循环**
```
- 最大 6 轮（普通模式）或 8 轮（思考模式）
- 每轮：模型返回 → 检查是否有 tool_calls → 有则执行 → 结果放回 → 继续
- 防止无限循环（模型可能反复调用工具）
```

#### 面试话术：
> "Agent 的核心是一个循环引擎。每次用户输入后，我把历史消息和工具定义一起发给模型，模型可能返回文本，也可能返回工具调用请求。如果它要调用工具，我就执行工具、把结果放回消息列表里，再调一次模型，让它基于工具结果生成最终回答。最多循环 6 轮防止死循环。"

> "流式事件我设计了一套统一的协议格式，前端不管什么类型的事件都按统一格式解析。这样后端可以灵活扩展事件类型，前端不需要跟着改。"

### 3.3 Provider 层（厂商 API 适配）

**文件：** [base.py](file:///f:/mini-agent/app/providers/base.py) + [openai_compat.py](file:///f:/mini-agent/app/providers/openai_compat.py)

#### 要讲的核心点：

**① 抽象基类设计**
```python
class BaseModelProvider(ABC):
    @abstractmethod
    def is_available(self) -> bool       # 检查 API Key 是否配置
    @abstractmethod
    def chat(self, ...) -> dict          # 非流式调用
    @abstractmethod
    def chat_stream(self, ...)           # 流式调用，yield 统一事件
```

**② 统一事件格式适配**
- 各厂商 API 返回格式不同（DeepSeek 有 reasoning_content，Kimi 的 tool_calls 格式不同）
- 在 `_parse_stream_delta()` 中统一转换为标准事件格式
- 上层（Agent）不关心底层是哪个厂商，只处理统一事件

**③ 重试机制**
```python
# 可重试的状态码：429（限流）、502/503/504（服务不可用）
_RETRYABLE_STATUS = {429, 502, 503, 504}
_MAX_RETRIES = 3

# 退避策略：指数退避 + 读 Retry-After 头
delay = _RETRY_BASE_DELAY * (2 ** attempt)
```

**④ 思考模式适配**
- 不同厂商思考模式参数不同：
  - DeepSeek: `thinking: {"type": "enabled"}`
  - Qwen/Kimi/GLM: `enable_thinking: true`
- 在 `apply_thinking_payload()` 中按厂商注入

#### 面试话术：
> "各厂商的 API 格式虽然都兼容 OpenAI 协议，但细节差异很大。比如 DeepSeek 的思考模式用 `thinking` 参数，Qwen 用 `enable_thinking`。我通过一个抽象基类统一接口，在 Provider 实现里做适配，上层 Agent 完全不用关心底层是哪个厂商。"

> "我还做了重试机制——遇到 429 限流或 502 宕机，自动重试最多 3 次，用指数退避策略。对于 Kimi K2 这类模型，temperature 必须设为 1.0，这些细节都在 Provider 层处理了。"

### 3.4 工具系统（Registry 注册模式）

**文件：** [registry.py](file:///f:/mini-agent/app/tools/registry.py) + [base.py](file:///f:/mini-agent/app/tools/base.py)

#### 要讲的核心点：

**① 注册模式**
```python
class ToolRegistry:
    def __init__(self):
        self._tools: dict[str, Tool] = {}  # 用字典维护

    def register(self, tool: Tool):         # 注册
    def get_definitions(self):              # 获取 OpenAI 格式的工具定义
    def execute(self, name, args):          # 执行
```

**② Tool 抽象基类**
```python
class Tool(ABC):
    name: str          # 工具名，也是唯一标识
    description: str   # 描述，供模型理解何时调用
    parameters: dict   # JSON Schema 格式的参数定义

    def to_definition(self) -> dict:  # 转为 OpenAI tools 格式
    def execute(self, args) -> str:   # 执行逻辑
```

**③ 已注册的工具列表**
| 工具名 | 功能 |
|--------|------|
| `list_files` | 列出 workspace 文件 |
| `read_file` | 读取文件内容 |
| `write_file` | 写入文件 |
| `search_file` | 按文件名搜索 |
| `calculator` | 安全计算表达式（AST 解析，非 eval） |
| `db_query` | 查询会话统计 |
| `rag_search` | 知识库语义检索 |
| `rag_index` | 文件索引到知识库 |
| `rag_status` | 知识库状态查询 |
| `web_search` | 预留的网页搜索 |
| `mcp_*` | 通过 MCP 协议接入的外部工具 |

**④ 安全计算器（重要细节）**
- 使用 `ast.parse()` 解析表达式，而不是 `eval()`
- 只允许白名单内的操作符（+ - * / // % **）
- 防止任意代码执行

#### 面试话术：
> "工具系统用了注册模式。每个工具都继承一个抽象基类，定义好名字、描述和参数 Schema。注册中心统一管理，需要获取所有工具定义时调用 `get_definitions()`，自动转成 OpenAI 的 tools 格式发给模型。"

> "安全方面，计算器我没有用 eval，而是用 AST 解析——只允许白名单里的数学操作符，从根本上杜绝了注入风险。文件操作也做了路径穿越防护，所有路径都限制在 workspace 目录内。"

### 3.5 RAG 知识库

**文件：** [store.py](file:///f:/mini-agent/app/rag/store.py) + [chunker.py](file:///f:/mini-agent/app/rag/chunker.py) + [embedder.py](file:///f:/mini-agent/app/rag/embedder.py)

#### 要讲的核心点：

**① 架构**
```
上传文件 → chunk_text() 分块 → RagStore.add_document() → 向量化 → 构建索引
用户提问 → rag_search() → 向量检索 → 返回相关片段 → 模型基于片段回答
```

**② 分块策略（`chunk_text`）**
- 优先按段落（`\n\n`）分割
- 块大小 500 字符，块间重叠 80 字符
- 超过块大小的段落再按长度切分
- 保证语义完整性

**③ 向量化方案（两套）**
- **默认 TF-IDF**：零依赖，轻量，适合小规模知识库
  - 使用 `sklearn.feature_extraction.text.TfidfVectorizer`
  - 最大 8000 特征，1-2 元词组
  - 余弦相似度计算
- **可选 FAISS + 稠密向量**：通过 `.env` 配置 `RAG_EMBEDDING_MODE=api` 或 `local`
  - API 模式：调用 embedding API（如 OpenAI text-embedding-3-small）
  - 本地模式：用 SentenceTransformer（如 paraphrase-multilingual-MiniLM-L12-v2）
  - FAISS 内积索引，支持大规模检索

**④ 自动索引**
- 上传文件时自动调用 `_index_to_rag()`
- 索引失败不阻塞主流程（try/except 吞异常）
- 支持增量更新：重新索引同一文件会先删旧数据

#### 面试话术：
> "RAG 知识库支持两套向量化方案。默认用 TF-IDF，不需要下载任何模型，适合小规模文档。如果需要更好的语义理解，可以切换到 API 或本地 embedding 模型，用 FAISS 做稠密检索。分块策略我考虑了语义完整性——优先按段落切，不是简单按字符数硬切。"

> "而且整个过程是自动化的——用户上传文件后自动分块、索引，不需要手动操作。Agent 在回答时如果检测到需要知识库内容，会调用 `rag_search` 工具来检索。"

### 3.7 前端 SSE 流式处理

**文件：** [chat.js](file:///f:/mini-agent/frontend/src/api/chat.js) + [App.vue](file:///f:/mini-agent/frontend/src/App.vue)

#### 要讲的核心点：

**① SSE 流式消费**
```javascript
// 读取 ReadableStream
const reader = res.body.getReader();
const decoder = new TextDecoder("utf-8");

// 处理粘包问题：用 \n\n 分割，buffer 缓存不完整的数据
while (true) {
    const { done, value } = await reader.read();
    buffer += decoder.decode(value, { stream: true });
    const parts = buffer.split("\n\n");
    buffer = parts.pop();  // 最后一个可能不完整
    for (const part of parts) {
        parseSSEChunk(part, callbacks);
    }
}
```

**② 粘包处理**
- SSE 事件以 `\n\n` 分隔
- 网络传输可能合并多个事件，也可能一个事件被拆成多次
- 用 buffer 累积，按 `\n\n` 分割，不完整的留在 buffer 里下次处理

**③ 事件解析**
```javascript
function parseSSEChunk(raw, callbacks) {
    // 每行以 "data:" 开头
    // JSON parse 后按 type 分发到不同回调
    switch (data.type) {
        case "thinking": onThinking(chunk);  break;
        case "token":    onToken(chunk);     break;
        case "status":   onStatus(chunk);    break;
        case "reset":    onReset();          break;
        case "done":     onDone(data);       break;
    }
}
```

**④ 中断机制**
- 使用 `AbortController` 实现停止生成
- 点击停止按钮 → `controller.abort()` → fetch 中断 → 标记消息为"已停止"

#### 面试话术：
> "前端流式处理有两个关键点。第一是粘包——SSE 事件以 `\n\n` 分隔，但网络传输可能合并或拆分，我用 buffer 累积数据，按分隔符切分，不完整的留在 buffer 里。第二是中断——用 AbortController 实现，用户点停止时立即中断 fetch，防止浪费 token。"

## 四、技术选型与设计决策（面试加分项）

### 4.1 为什么选 SSE 而不是 WebSocket？

| 维度 | SSE | WebSocket |
|------|-----|-----------|
| 协议 | HTTP 长连接 | 独立 TCP 连接 |
| 方向 | 服务器→客户端单向 | 双向 |
| 复杂度 | 低，原生支持 | 高，需要握手/心跳 |
| 兼容性 | 原生 EventSource API | 需要额外库 |
| 重连 | 浏览器自动重连 | 需手动实现 |
| 适用场景 | 流式输出（正好是我们需要的） | 实时双向通信 |

**结论：** 我们的场景是模型→用户的单向流式输出，不需要双向通信。SSE 更轻量、更简单，浏览器原生支持自动重连，完全够用。

### 4.2 为什么默认用 TF-IDF 而不是深度学习向量？

- **零依赖**：不需要下载 embedding 模型，不需要 GPU
- **启动快**：项目 clone 下来就能跑
- **小规模够用**：几百个文档的规模，TF-IDF + 余弦相似度效果不错
- **可扩展**：.env 配置 `RAG_EMBEDDING_MODE=api` 或 `local` 即可切换到稠密向量

### 4.3 为什么用 SQLite 而不是 Redis/MySQL？

- **零运维**：不需要额外安装数据库服务
- **单机够用**：个人使用的场景，没有并发问题
- **文件级备份**：直接备份 `.db` 文件即可
- **SQL 灵活**：比 JSON 文件查询更灵活，支持复杂统计

### 4.4 为什么 Agent 循环最大 6-8 轮？

- 防止模型陷入无限工具调用循环
- 大多数场景 1-2 轮工具调用就够
- 思考模式允许更多轮（8 轮），因为思考链可能更长

## 五、面试可能被问到的问题（预判与应答）

### Q1: 这个项目你主要负责什么？

**不要这样说：** "我用 Cursor 生成的，我就改改报错。"

**要这样说：**
> "这个项目是我主导设计的，我负责整体架构设计、核心模块编写和技术选型。具体来说：
> 1. 设计了多模型路由架构，抽象出统一的 Provider 接口，让团队可以快速接入新模型
> 2. 实现了 Agent 主循环引擎，包括消息构建、流式事件协议、工具调用循环
> 3. 搭建了 RAG 知识库，设计了分块策略和可切换的向量化方案
> 4. 前后端联调，处理了 SSE 流式输出的一系列细节问题（粘包、中断、事件格式统一）
> 5. 做了容错处理：API 重试、模型不可用时的兜底策略、思考模式降级等"

### Q2: 项目中遇到的最大挑战是什么？

**建议回答：**
> "最大的挑战是流式事件的统一。各厂商 API 返回格式差异很大——DeepSeek 有 reasoning_content，Kimi 的 tool_calls 格式不同，有的厂商支持思考模式有的不支持。我需要设计一套统一的事件协议，让 Agent 层和前端都不关心底层是哪个厂商。
>
> 解决方案是三层抽象：
> 1. Provider 层：把各厂商的 API 响应统一为标准事件
> 2. Agent 层：只处理标准事件，不关心厂商差异
> 3. 前端层：按 type 字段分发，未来加新事件类型不用改解析逻辑
>
> 另一个挑战是工具调用循环的稳定性。模型可能连续调用多个工具，也可能在一个工具上反复调用。我设置了最大循环轮次，并且在工具执行出错时不会崩溃整个流程，而是把错误信息返回给模型让它自己处理。"

### Q3: 为什么选择 FastAPI 而不是 Flask/Django？

**建议回答：**
> "选择 FastAPI 有几个原因：
> 1. 异步原生支持——SSE 流式输出需要异步生成器，FastAPI 的 StreamingResponse 天然支持
> 2. 性能——基于 Starlette，性能接近 Node.js/Go
> 3. 自动文档——自动生成 OpenAPI 文档，调试方便
> 4. 类型校验——Pydantic 集成，请求参数自动校验
>
> 如果用 Flask，SSE 需要额外处理，而且 Flask 的同步模型在处理长连接时会占用 worker 线程。"

### Q4: 如果用户量大了，这个系统哪里会成为瓶颈？

**建议回答：**
> "目前设计是单机单进程，几个瓶颈点：
> 1. **Agent 实例内存占用**——每个会话持有一个 Agent 实例，包含完整对话历史，会话多了内存压力大。可以加 LRU 淘汰策略。
> 2. **SQLite 并发**——SQLite 写操作是串行的，高并发下会成瓶颈。需要迁移到 PostgreSQL。
> 3. **RAG 检索**——TF-IDF 是内存计算，文档多了速度下降。可以切换到专门的向量数据库（Milvus/Pinecone）。
> 4. **API 调用**——目前是同步 requests，多个请求会阻塞。可以改为异步 httpx 或加队列。
>
> 但这些对当前场景（个人使用、小规模）不是问题，设计时考虑了可替换性——比如 RAG 从 TF-IDF 切换到 FAISS 只需要改一行配置。"

### Q5: 多个模型切换时，如何保证对话连贯性？

**建议回答：**
> "切换模型时，对话历史是完整的——所有消息都保存在 SQLite 中。切换后重建 Agent 时，会从数据库加载完整的历史消息，包括之前的对话和工具调用结果。所以用户切换模型后，新模型能"看到"完整的上下文。
>
> 不过有一个问题：不同模型的 system prompt 可能不同，切换后 system prompt 会变，但历史消息不变。这是个权衡——如果要完美一致，需要每个模型保存独立的对话历史，但那样切换模型就失去了意义。"

### Q6: 你了解 MCP 协议吗？项目里怎么用的？

**建议回答：**
> "MCP（Model Context Protocol）是一个开放协议，标准化的方式让 AI 模型调用外部工具。项目里通过 MCP 客户端桥接，可以连接任何 MCP Server。
>
> 实现方式：
> 1. MCPBridge 在后台线程维护 MCP 连接
> 2. 用 `ClientSessionGroup` 管理多个 MCP Server 连接
> 3. 从 MCP Server 拉取工具列表，转换为 OpenAI 兼容格式注入 ToolRegistry
> 4. Agent 调用工具时，MCPBridge 将请求转发到对应的 MCP Server
>
> 这个设计让项目可以接入任意外部工具，比如数据库查询、API 调用、文件系统操作等，而不需要自己实现每个工具。"

### Q7: 如果模型 API 调用失败了，系统怎么处理？

**建议回答：**
> "做了多层容错：
> 1. **重试机制**：429 限流、502/503/504 宕机，自动重试 3 次，指数退避
> 2. **模型降级**：如果用户选的模型不可用，自动切换到第一个可用的模型
> 3. **思考模式降级**：如果用户开启思考模式但模型不支持，自动降级为普通模式并提示用户
> 4. **工具执行容错**：工具执行失败不崩溃，错误信息返回给模型让它处理
> 5. **RAG 索引容错**：索引失败不影响聊天流程
>
> 前端侧也做了处理：AbortController 实现中断，网络错误时显示友好的错误提示。"

### Q8: 这个项目可以怎么扩展？

**建议回答：**
> "扩展方向：
> 1. **模型层**：加新厂商只需写一个 Provider 类，接进来就行
> 2. **工具层**：实现 Tool 抽象类，调用 register() 即可添加新工具
> 3. **RAG 层**：从 TF-IDF 切换到稠密向量只需改配置
> 4. **MCP 协议**：可以接入任意 MCP Server，扩展性很强
> 5. **前端**：可以加语音输入、图片理解等
> 6. **部署**：可以加 Docker 化、Nginx 反代、多 worker 等"
